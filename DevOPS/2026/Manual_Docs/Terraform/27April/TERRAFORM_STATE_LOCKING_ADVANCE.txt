🔶 GLOBAL PRE-CAUTIONS ⚠
----------------------------------
- Ensure Azure login using "az login" is completed successfully 🔐
- Decide naming conventions before creating any Azure resource 📌
- Storage account name must be lowercase and globally unique 🌍
- Backend infrastructure must exist before running terraform init ⚙
- Avoid multiple users running terraform apply simultaneously ❌
- Always run terraform plan before apply to validate changes 👀

🔷 SECTION 2 — CREATE BACKEND INFRA (AZURE CLI ☁)
----------------------------------

az login

az storage account create \
  --name myappdevst001 \
  --resource-group rg-dev-storage \
  --location centralindia \
  --sku Standard_LRS

az storage container create \
  --name tfstate \
  --account-name myappdevst001

DETAILS:
- Resource group logically groups backend infrastructure resources together 📦
- Storage account stores Terraform state file as blob internally 📄
- Container acts as folder where Terraform state file will reside 📁
- Backend must exist before Terraform initialization step ⚠
- Enables remote state storage and locking mechanism 🔒
- Required for team collaboration and CI/CD workflows 🚀

PRE-CAUTION:
- Ensure storage account name is globally unique before creation ✔

POST-CAUTION:
- Verify resources using Azure CLI after creation ✔

----------------------------------
🔷 SECTION 3 — BACKEND CONFIG (backend.tf ⚙)
----------------------------------

terraform {
  backend "azurerm" {
    resource_group_name  = "rg-dev-storage"
    storage_account_name = "myappdevst001"
    container_name       = "tfstate"
    key                  = "dev.terraform.tfstate"
  }
}

DETAILS:
- Backend block defines remote location where Terraform state will be stored 📄
- key defines blob file name used for storing state file 📌
- Enables Terraform to connect to Azure Blob storage 🌐
- Automatically enables locking using Blob lease mechanism 🔒
- Must be configured before running terraform init ⚠
- Uses terraform block with backend "azurerm" type ⚙

PRE-CAUTION:
- Ensure backend values exactly match Azure resources ✔

POST-CAUTION:
- Do not change backend after resources are already created ❌

----------------------------------
🔷 SECTION 4 — MAIN INFRASTRUCTURE (main.tf 🏗)
----------------------------------

provider "azurerm" {
  features {}
}

resource "azurerm_resource_group" "rg" {
  name     = "rg-dev-storage"
  location = "Central India"
}

resource "azurerm_storage_account" "sa" {
  name                     = "myappdevst001"
  resource_group_name      = azurerm_resource_group.rg.name
  location                 = azurerm_resource_group.rg.location
  account_tier             = "Standard"
  account_replication_type = "LRS"
}

resource "azurerm_storage_container" "container" {
  name                  = "app-data"
  storage_account_name  = azurerm_storage_account.sa.name
  container_access_type = "private"
}

DETAILS:
- Terraform automatically builds dependency graph using resource references 🔗
- Resource group must exist before storage account creation ⚙
- Storage account must exist before container creation 📦
- No manual depends_on required in this scenario ✔
- Terraform executes resources in correct order using DAG internally 🔄
- Code remains declarative and easy to maintain 📘

PRE-CAUTION:
- Ensure naming rules especially for storage account are followed ⚠

POST-CAUTION:
- Verify resources using Azure CLI or portal after apply ✔

🔷 SECTION 5 — TERRAFORM EXECUTION FLOW ⚙
----------------------------------

terraform init  
   ➜ Connect backend  
   ➜ Prepare state location  

terraform plan  
   ➜ Read state  
   ➜ Compare with code  

terraform apply  
   ➜ LOCK 🔒  
   ➜ CREATE RESOURCES  
   ➜ UPDATE STATE  
   ➜ UNLOCK 🔓  

DETAILS:
- init downloads provider plugins and configures backend connection 🔌
- plan shows execution plan without making changes 🔍
- apply performs actual resource creation or modification ⚙
- Locking ensures safe execution without conflicts 🔒
- State updated after successful execution 📄
- Unlock happens automatically after completion 🔓

PRE-CAUTION:
- Always review plan output carefully before applying ✔

POST-CAUTION:
- Validate created resources and state consistency after execution ✔
