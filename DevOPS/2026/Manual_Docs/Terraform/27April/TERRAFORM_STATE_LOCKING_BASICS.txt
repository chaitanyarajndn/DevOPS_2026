========================================
🚀 TERRAFORM STATE & LOCKING (ABSOLUTE BASICS)
========================================

----------------------------------
🔶 PRE-CAUTIONS (BEFORE STARTING)
----------------------------------
- Always treat Terraform state as single source of truth for infrastructure 📌
- Never manually edit or delete state file unless you fully understand impact ⚠
- Ensure Azure login using "az login" is completed successfully before execution 🔐
- Prefer remote backend over local state for safety and team collaboration 🌐
- Avoid running terraform apply from multiple systems simultaneously ❌
- Always run terraform plan first to understand upcoming infrastructure changes 👀

----------------------------------
🔷 1. AZURE STORAGE BASICS ☁
----------------------------------
Storage Account ➜ Container ➜ Blob (File)

Explanation:
- Storage Account acts as globally unique cloud storage service endpoint 🌍
- Container acts like folder grouping related files logically inside storage 📁
- Blob is actual file where Terraform state is stored and updated 📄

Example:
myappdevst001 ➜ tfstate ➜ dev.terraform.tfstate

DETAILS:
- Storage account must be lowercase and globally unique across Azure systems ⚠
- Blob is accessed via HTTP endpoint, so naming follows DNS rules strictly 🌐
- Terraform continuously reads and writes state file during operations 🔄
- Container helps separate environments like dev, prod, and staging clearly 🧩
- Backend configuration connects Terraform to this Azure storage structure 🔗

PRE-CAUTION:
- Ensure storage account naming rules are followed before creation ✔

POST-CAUTION:
- Never delete storage account or container containing Terraform state ❌

----------------------------------
🔷 2. TERRAFORM STATE (CORE MEMORY 🧠)
----------------------------------
Terraform Code ➜ State File ➜ Azure Resources

Explanation:
- State file stores current infrastructure details created by Terraform 📊
- Includes resource IDs, metadata, and configuration snapshot 📦
- Terraform compares desired state with actual state during execution 🔍

DETAILS:
- Without state file, Terraform assumes nothing exists and recreates everything ❌
- State is used internally in terraform plan and apply operations 🔄
- Sensitive data like secrets or keys may be present inside state 🔐
- State must be securely stored using backend and proper access control 🛡
- Remote state enables team collaboration and consistent infrastructure management 🤝

PRE-CAUTION:
- Never expose state file publicly due to sensitive data risk ⚠

POST-CAUTION:
- Always verify state using "terraform state list" after apply ✔

----------------------------------
🔷 3. LOCAL vs REMOTE STATE (IMPORTANT 🔥)
----------------------------------

LOCAL STATE (terraform.tfstate on laptop)
✘ Not shared across team members
✘ No locking mechanism available
✘ High risk of corruption and overwriting
✘ Lost if system crashes or file deleted
✘ Not suitable for production usage

REMOTE STATE (Azure Blob)
✔ Centralized storage accessible by entire team
✔ Locking enabled using Azure Blob lease mechanism 🔒
✔ Safe for CI/CD pipelines and automation workflows 🚀
✔ Durable and backed by Azure reliability guarantees ☁
✔ Secure with Azure IAM and role-based access control 🔐

----------------------------------
🔷 4. STATE LOCKING (MAIN CONCEPT 🔒)
----------------------------------
User A Apply ➜ LOCK 🔒 ➜ Work ➜ UNLOCK 🔓  
User B Apply ➜ BLOCKED ❌ ➜ ERROR

Explanation:
- Locking ensures only one Terraform operation modifies state at a time ⚙
- Prevents conflicts, corruption, and inconsistent infrastructure creation ❌
- Automatically handled when using remote backend like Azure Blob 🌐

DETAILS:
- Lock is acquired before Terraform starts modifying infrastructure 🔐
- Lock is released after Terraform completes execution successfully 🔓
- Other users attempting apply will receive state lock error ❌
- Lock ensures consistency between state file and real infrastructure 🔄
- Essential for team collaboration and CI/CD environments 🤝

PRE-CAUTION:
- Never ignore lock errors or try bypassing locking manually ❌

POST-CAUTION:
- Confirm no active process before using force-unlock command ⚠

----------------------------------
🔷 5. HOW LOCKING WORKS (AZURE LEASE 🔑)
----------------------------------

terraform apply  
   ➜ Request Lock  
      ➜ Azure grants Lease 🔒  
         ➜ Terraform updates infrastructure ⚙  
            ➜ State file updated 📄  
               ➜ Lease released 🔓  

DETAILS:
- Azure Blob lease provides exclusive access to state file 🔐
- Only one lease can exist at any given time on blob 📌
- Terraform manages lease lifecycle automatically without manual setup 🤖
- Lease exists only during Terraform operation execution ⏱
- Prevents concurrent modification and ensures data consistency 🔄
- No additional configuration required for enabling locking ✔

PRE-CAUTION:
- Ensure backend is configured correctly or locking will not work ❌

POST-CAUTION:
- Monitor failed executions as abrupt termination may leave stale locks ⚠

----------------------------------
🔷 6. LOCK CONFLICT (REAL SCENARIO ⚔)
----------------------------------
User A Apply ➜ LOCKED 🔒  
User B Apply ➜ TRY LOCK ➜ FAIL ❌ ➜ ERROR

DETAILS:
- Terraform checks lock existence before starting execution 🔍
- If lock exists, operation fails immediately to prevent conflict ❌
- Error includes lock details such as user and timestamp 📌
- Common in teams and CI/CD pipelines running parallel jobs 🤝
- Ensures infrastructure consistency and prevents corruption 🔄

PRE-CAUTION:
- Coordinate with team before running terraform apply operations 🤝

POST-CAUTION:
- Investigate lock owner before using force-unlock command ⚠
