Tool: Terraform (Infrastructure as Code)

---

## 📌 1. DEFINITION

✔ Definition

In Terraform, dependency defines the order in which resources are created, updated, or destroyed.

✔ Meaning	- **🔷 TERRAFORM DEPENDENCY (CORE UNDERSTANDING)**


* Dependency defines the **execution order of resources** (create, update, destroy) in Terraform

* Terraform builds a **Directed Acyclic Graph (DAG)** where:

  * Nodes = resources
  * Edges = dependencies

* Dependencies are primarily created via **resource attribute references**
  (`resource_type.name.attribute`)

* Terraform follows **data flow**, not logical assumptions
  👉 If no reference → no dependency (even if logically required)

* Resources without dependencies are executed **in parallel** to optimize performance

* Dependency ensures that a resource is created **only after its upstream resource is fully ready**

* There are two ways to define dependency:

  * **Implicit** → via references (recommended)
  * **Explicit** → via `depends_on` (manual control)

* `depends_on` adds **manual edges to DAG**, forcing execution order even without references

* Overusing explicit dependencies can **reduce parallelism and slow down execution**

---

**🎯 ONE-LINE INTERVIEW ANSWER**

👉 *“Terraform dependency is managed through a DAG, where execution order is determined by resource references (implicit) or manually enforced using depends_on (explicit), ensuring correct and optimized provisioning.”*


👉 Example:
A Virtual Machine needs a Network Interface → NIC must be created first.


👉 Simple Meaning:
If Resource A needs Resource B → B must be created first

Example:
VM → NIC → Subnet → VNet

## 📌 1. DEFINITION

DAG = Directed Acyclic Graph

👉 In Terraform, DAG is the internal structure used to:

* Manage dependencies
* Decide execution order

---

## 📌 2. BREAKDOWN (IMPORTANT)

🔹 Directed

* Each dependency has a direction
* Example:
  Resource A -> Resource B

👉 Meaning:
B depends on A (A runs first)

---

🔹 Acyclic

* No circular dependency allowed

❌ Invalid:
A -> B -> C -> A

👉 Terraform will throw error

---

🔹 Graph

* Collection of:
  Nodes = Resources
  Edges = Dependencies

---

## 📌 3. HOW TERRAFORM USES DAG

Step 1: Read configuration
Step 2: Detect dependencies (implicit + explicit)
Step 3: Build DAG
Step 4: Execute based on graph

---

## 📌 4. SIMPLE FLOW EXAMPLE

[Resource Group]
↓

👉 Each arrow = dependency
👉 Full structure = DAG

## 📌 5. EXECUTION BEHAVIOR

✔ No dependency → runs in parallel

✔ With dependency → runs in sequence

👉 Terraform optimizes execution using DAG

## 📌 6. WHY DAG IS IMPORTANT

🔥 DAG FLOW – SIMPLE EXAMPLE (RESOURCE GROUP → STORAGE ACCOUNT) 🔥

---

## 📌 1. SCENARIO

We want to create:

* Resource Group
* Storage Account

👉 Storage Account depends on Resource Group

---

## 📌 2. TERRAFORM CODE

resource "azurerm_resource_group" "rg" {
name     = "demo-rg"
location = "East US"
}

resource "azurerm_storage_account" "sa" {
name                     = "demostorage123"
resource_group_name      = azurerm_resource_group.rg.name
location                 = azurerm_resource_group.rg.location
account_tier             = "Standard"
account_replication_type = "LRS"
}

👉 Dependency created via:

* rg.name
* rg.location

---

## 📌 3. DAG FLOW (VISUAL)

[Resource Group]
↓
[Storage Account]

👉 Meaning:

* First create Resource Group
* Then create Storage Account

---

## 📌 4. HOW TERRAFORM THINKS

Step 1: Reads both resources

Step 2: Finds reference:
azurerm_resource_group.rg.name

Step 3: Builds DAG:

RG → Storage Account

Step 4: Executes:

1. Create RG
2. Create Storage Account

---

## 📌 5. PARALLEL VS SEQUENTIAL

✔ Resource Group → must run first

✔ Storage Account → waits for RG

👉 Sequential execution (due to dependency)

---

## 📌 6. IF NO REFERENCE (IMPORTANT)

❌ Example (Wrong):

resource_group_name = "demo-rg"

👉 No dependency created

👉 Terraform may run both in parallel

👉 Storage account creation may fail

---

## 📌 7. EXPLICIT VERSION (FOR UNDERSTANDING)

resource "azurerm_storage_account" "sa" {
name = "demostorage123"

depends_on = [
azurerm_resource_group.rg
]
}

👉 Forces same flow:

RG → Storage Account

---

## 📌 8. FINAL UNDERSTANDING

✔ Reference present → Implicit dependency

✔ No reference → No dependency

✔ depends_on → Forced dependency

👉 DAG ensures correct order every time

★══════════════════════════════════════════★
★    ✨✨  IMPLICIT DEPENDENCY  ✨✨    ★
★══════════════════════════════════════════★

📖 Definition:
Terraform automatically detects dependency using references

🧠 Meaning (DETAILED UNDERSTANDING):

* Created when a resource **references another resource’s attribute** (id, name, location, etc.)

* Based on **expression evaluation in HCL**, not manual instruction

* When you use `resourceA.attribute` → Terraform creates a **dependency edge in DAG**

* Terraform builds a **Directed Acyclic Graph (DAG)** internally to determine execution order

* Ensures correct **create, update, and destroy sequence**

* Allows **parallel execution** of unrelated resources → improves performance

* Dependency is **visible through references**, not explicitly declared

* Removing reference → automatically removes dependency (dynamic behavior)

* Guarantees that dependent resource waits until **source resource is fully created**

* Best suited for **standard infra chaining**
  (VNet → Subnet → NIC → VM)

---

## ✨ CHARACTERISTICS

✔ Automatic (no manual effort)
✔ Based on resource reference
✔ Clean & readable
✔ Preferred approach
✔ Maintains parallel execution

---

## 💻 CODE FORMAT (DEEP UNDERSTANDING)

resource "azurerm_resource_group" "rg" {
name     = "demo-rg"
location = "East US"
}

resource "azurerm_virtual_network" "vnet" {
name                = "demo-vnet"
location            = azurerm_resource_group.rg.location
resource_group_name = azurerm_resource_group.rg.name
}

👉 Dependency created because:
azurerm_resource_group.rg.name
azurerm_resource_group.rg.location

🔍 HOW DEPENDENCY IS FORMED:

* Terraform scans for ANY reference:
  azurerm_resource_group.rg.*

* Each reference creates dependency edge in DAG

🧠 IMPORTANT RULE:

✔ ANY attribute reference creates dependency:

* name
* id
* location
* tags
* any exported attribute

👉 Attribute TYPE does NOT matter
👉 Only REFERENCE matters

---

## 📌 MORE AZURE EXAMPLES

✔ Using NAME
resource_group_name = azurerm_resource_group.rg.name

✔ Using LOCATION
location = azurerm_resource_group.rg.location

✔ Using TAGS
tags = azurerm_resource_group.rg.tags

✔ Using ID (very common)
subnet_id = azurerm_subnet.subnet.id

✔ Using list reference
network_interface_ids = [
azurerm_network_interface.nic.id
]

👉 ALL above create implicit dependency

---

## 🌐 WHERE TO VERIFY ATTRIBUTES

👉 Terraform Registry (Provider Docs)

Search resource (example: azurerm_virtual_network)

Check:

* Arguments (inputs)
* Attributes (outputs you can reference)

👉 Dependency comes from:
Attributes you reference

❌ Terraform does NOT understand logic
✔ It ONLY understands references

👉 No reference = No dependency

Example:

# no reference to A

}

➡️ Terraform may run both in parallel

★══════════════════════════════════════════★
★    ✨✨  EXPLICIT DEPENDENCY  ✨✨    ★
★══════════════════════════════════════════★

📖 Definition:
Manually defining dependency using:

depends_on = []

🧠 Meaning (DETAILED UNDERSTANDING):

* Defined manually using `depends_on` when Terraform **cannot infer dependency**

* Adds a **manual dependency edge in DAG**

* Forces Terraform to **respect execution order regardless of references**

* Useful for **hidden dependencies**
  (scripts, APIs, provisioners, external setup)

* Ensures one resource completes **before another starts**

* Common with:
  null_resource, local-exec, remote-exec

* Can override Terraform’s default behavior

* May reduce **parallel execution**, impacting performance

* Overuse leads to **tight coupling and harder maintenance**

* Should be used only when **implicit dependency is not sufficient**

---

## ✨ CHARACTERISTICS

✔ Manual control
✔ Used in special cases
✔ Overrides Terraform behavior

---

## 💻 CODE FORMAT (DEEP UNDERSTANDING)

resource "azurerm_virtual_machine" "vm" {
name = "demo-vm"

depends_on = [
azurerm_network_interface.nic,
azurerm_subnet.subnet
]
}

🔍 HOW THIS WORKS:

👉 You are explicitly telling Terraform:

"Create NIC and Subnet FIRST, then VM"

👉 Even WITHOUT attribute reference → dependency exists

---

## 📌 SPECIAL CASE

depends_on = [
module.network
]

👉 Works with:

* resources
* modules
* data sources

---

## 📌 REAL SCENARIOS

🔹 Script execution first

resource "null_resource" "script" {
provisioner "local-exec" {
command = "echo Setup Complete"
}
}

resource "azurerm_virtual_machine" "vm" {
name = "demo-vm"

depends_on = [null_resource.script]
}

👉 VM waits for script execution

🔹 External dependencies
✔ APIs
✔ Provisioners
✔ Manual configurations

## 🎯 6. WHEN TO USE WHAT

✅ Use Implicit:
✔ When resource reference exists
✔ Standard infrastructure
✔ Clean architecture

👉 Example:
VM → NIC → Subnet → VNet

---

✅ Use Explicit:
✔ No direct reference
✔ Script/API dependency
✔ Hidden dependency exists

👉 Example:
Script → then VM

---

## ⚠️ 7. IMPORTANT INSIGHT

👉 Terraform uses DAG (Dependency Graph)

✔ Implicit → Data flow driven
✔ Explicit → Control flow enforced

🚨 Warning:
Too much depends_on = ❌ kills parallel execution

---

## 🏁 8. FINAL SUMMARY

👉 Implicit = Smart Terraform
👉 Explicit = You guiding Terraform

✔ Best Practice:
Use Implicit (90% cases)
Use Explicit (only when required)
