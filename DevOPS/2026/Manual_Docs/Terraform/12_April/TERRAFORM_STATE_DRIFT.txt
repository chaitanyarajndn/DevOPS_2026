==================== TERRAFORM_STATE_DRIFT NOTES ====================

Purpose:
Understand the relationship and behavior between:

* .tf file        → Desired State (WHAT you want)
* .tfstate file   → Known State (WHAT Terraform tracks)
* Azure Portal    → Actual Infrastructure (WHAT really exists)

Terraform Objective:
→ Ensure Desired State = Actual Infrastructure
→ Achieve and maintain "Equilibrium" (Desired State Convergence)

---

## CORE CONCEPTS: DRIFT, RECONCILIATION & EQUILIBRIUM

1. Infrastructure Drift:
   Drift occurs when:
   → Azure (Actual Infra) ≠ .tfstate (Terraform’s Known State)

Examples:

* Resource deleted manually
* Resource modified outside Terraform

Impact:
→ Terraform’s understanding becomes incorrect

---

2. State Reconciliation:
   Process by which Terraform fixes mismatches

Steps:

* terraform refresh → Sync state with Azure
* terraform plan    → Compare Desired (.tf) vs Current (.tfstate)
* terraform apply   → Fix differences

---

3. Equilibrium (Desired State Convergence):
   System is in equilibrium when:
   → .tf = .tfstate = Azure

Meaning:

* No pending changes
* No drift
* Fully aligned infrastructure

---

SCENARIO 1:
Resource present ONLY in .tf file
(NOT in .tfstate and NOT in Azure)
----------------------------------

Situation:

* Resource is defined in configuration (.tf)
* Resource is NOT present in .tfstate
* Resource does NOT exist in Azure

(This is NORMAL — new resource creation case)

What Happens:

* terraform plan  → Shows resource as "to be created"
* terraform apply → Creates resource in Azure
* Terraform updates .tfstate with this resource

Professional Explanation:
Terraform compares Desired State (.tf) with Known State (.tfstate) and Actual Infrastructure (Azure). Since the resource exists only in the configuration and nowhere else, Terraform identifies it as missing infrastructure. It provisions the resource and records it in the state file, moving the system toward equilibrium (convergence).

Key Point:
→ Exists only in .tf = Terraform will CREATE it

Precautions:

* Validate configuration (`terraform validate`)
* Check naming conflicts in Azure
* Always review `terraform plan`
* Follow consistent naming standards

Post-Cautions:

* Verify resource in Azure Portal
* Ensure .tfstate is updated correctly (remote backend recommended)
* Avoid manual changes immediately after creation
* Confirm tagging, policies, and dependencies

---

SCENARIO 2:
Resource present in .tf AND .tfstate BUT NOT in Azure
-----------------------------------------------------

Situation:

* Resource is defined in .tf
* Resource is tracked in .tfstate
* Resource is missing in Azure (deleted manually / external change)

(This is TRUE DRIFT scenario)

What Happens:

* terraform refresh → Detects resource missing in Azure
* terraform plan    → Marks resource for RE-CREATION
* terraform apply   → Recreates resource in Azure
* .tfstate updated accordingly

Professional Explanation:
Terraform detects Infrastructure Drift because the actual infrastructure does not match its known state. Even though Terraform believes the resource exists (based on .tfstate), Azure does not have it. Through State Reconciliation, Terraform recreates the resource to restore alignment and achieve Desired State Convergence (equilibrium).

Key Point:
→ Drift detected = Terraform RECONCILES by recreating resource

Precautions:

* Avoid manual deletion/modification in Azure
* Enforce Infrastructure as Code (IaC) discipline
* Use remote state backend with locking
* Restrict direct access to production resources

Post-Cautions:

* Resource IDs/IPs may change after recreation
* Validate dependent resources (networking, IAM, etc.)
* Check for downtime or service impact
* Carefully review `terraform plan` before apply

---

SCENARIO 3:
Resource present ONLY in Azure
(NOT in .tf AND NOT in .tfstate)
--------------------------------

Situation:

* Resource exists in Azure Portal
* Resource is NOT defined in .tf
* Resource is NOT tracked in .tfstate

(This is UNMANAGED resource — NOT drift)

What Happens:

* Terraform does NOTHING
* `terraform plan` shows no changes
* Resource remains untouched

Professional Explanation:
Terraform is unaware of this resource because it is neither part of the Desired State nor tracked in the Known State. Since Terraform operates only on declared and tracked resources, it ignores this completely. No reconciliation occurs because Terraform has no reference to compare.

Key Point:
→ Not in .tf and not in state = Terraform IGNORES it

Precautions:

* Avoid manual resource creation outside Terraform
* Maintain strict IaC practices
* Perform regular Azure audits
* Keep environments consistent across stages

Post-Cautions:

* Use `terraform import` if resource should be managed
* Document unmanaged resources clearly
* Risk of duplication if later defined in .tf
* Ensure naming alignment before importing

          Desired State        Known State        Actual Infra
              (.tf)            (.tfstate)          (Azure)
                 \                 |                 /

                   \_____ RECONCILIATION _________/

Terraform Lifecycle:

1. Refresh → Sync state with Azure
2. Plan    → Compare desired vs current

## IMPORTANT DISTINCTIONS

Scenario 1 → No Drift (New Resource Creation)
Scenario 2 → Drift Detected (Reconciliation Happens)
Scenario 3 → No Drift (Unmanaged Resource)
